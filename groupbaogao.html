壹、製作動機

（一）創作動機

本作的靈感源自於本身在連鎖牛丼店的真實打工經歷。面對繁重的出餐壓力和死板的公司制度，我們希望透過遊戲化的方式，將這種「崩潰但必須合作」的體驗轉化為樂趣。起初雖帶有對前公司的一種幽默「報復」心態，但最終目標是製作一款能考驗友情、強調溝通的合作遊戲。

（二）遊戲目的與核心價值

《美味牛丼》不僅僅是模擬餐廳經營，更是一款強調「多人即時協作」的派對遊戲。透過將廚房工序拆解為五個高度相依的環節，強迫玩家必須開口溝通（甚至吵架）。遊戲旨在讓玩家體驗在混亂中建立秩序的成就感，同時以詼諧的卡通風格呈現餐飲業的辛酸與趣味。

貳、遊戲基本介紹與定位

（一）遊戲定位

  • 遊戲類型： 多人合作模擬經營 / 單人模擬經營
  • 平台： 行動裝置
  • 遊玩人數： 5人連線 / 1人單機遊玩
  • 關鍵字： #模擬經營 #策略 #獨立遊戲 #多人 #溝通 #友盡

（二）遊戲優勢

不同於市面上單人或雙人的烹飪遊戲（如《Cooking Mama》），本作採用獨特的「流水線分工機制」。五名玩家缺一不可，任何一個環節的停滯都會導致全線崩潰。這種設計參考了《Overcooked!》的混亂感與《Space Team》的溝通需求，創造出高強度的社交互動體驗。

參、遊戲故事背景

（一）故事大綱

主角群原為知名「連鎖牛丼飯店」的資深員工。然而，總公司新上任的 CEO 頒布了極不合理的規章制度，壓榨員工且忽視餐點品質。出於對美食的堅持與對體制的反抗，主角決定憤而離職，帶領一群夥伴「自己單幹」。他們結合過去所學的經營知識，創立了全新的品牌，誓言要用真正的美味與效率擊敗前東家。

肆、遊戲玩法與流程設計

（一）核心玩法機制

  • 多人版：遊戲將牛丼製作過程拆解為五個站點，分別由五名玩家即時連線操作。玩家必須透過滑動、點擊與長按來完成指令，並將半成品「傳送」給下一位玩家。
  • 單人版：在單人模式中，移除了「後勤補貨（玩家 E）」的機制，所有食材視為無限供應或自動補給，讓玩家能專注於製作流程。玩家需獨自操作原本由多人分擔的四個工作站，考驗玩家的記憶力與多工處理能力。

（二）多人版模式角色職位詳解

1. 玩家 A（外場/點單）：資訊中樞

  • 任務： 接收 NPC 客人的訂單，並口頭傳達給內場（如：「兩碗中牛丼、一杯可樂！」）。最後負責將 D 玩家做好的套餐送給客人並結帳。
  • 操作： 點擊接單、確認餐點正確性、滑動出餐。

2. 玩家 B（盛飯/肉鍋）：基礎奠基

  • 任務： 根據 A 的指令製作對應大小的白飯（小/中/大）以及加上對應大小的肉量。
  • 機制： 飯鍋有容量限制，飯用完時需大喊請求 E 玩家補飯。
  • 操作： 點擊製飯機按鈕、滑動傳送給 C。

3. 玩家 C（加料）：核心加工

  • 任務： 接收 B 的白飯，添加配料（蔥花、起司、泡菜、秋葵）。
  • 機制： 四種食材會隨時間消耗，見底前需請求 E 玩家補料。
  • 操作： 點擊配料、滑動傳送給 D。

4. 玩家 D（套餐組合）：最終出貨

  • 任務： 接收 C 的牛丼，依訂單準備飲料（可樂）與附餐（味噌湯/小菜），組合成完整托盤。
  • 機制： 飲料與碗盤有庫存限制，需請求 E 玩家補給。
  • 操作： 拖曳組合、滑動回傳給 A。

5. 玩家 E（後勤/備料）：遊走支援

  • 任務： 全場最忙碌的角色。需監聽 B、C、D 的需求，親自將食材「運送」到該玩家的畫面中。
  • 機制： 煮飯有冷卻時間（CD），需預判米飯用量，避免全場斷糧。
  • 操作： 切換視角、點擊補貨、控制煮飯時間。

（三）單人挑戰模式詳解

玩家需要在以下四個視窗/工作站之間快速切換：

  1. 外場/點單：當客人進入店內時，玩家需點擊客人接單，玩家需記住客人的特殊要求。最後將完成好的餐點提供給 NPC 客人。
  2. 盛飯/肉鍋：製作對應大小的白飯（小/中/大）以及加上對應大小的肉量。
  3. 加料：添加配料（蔥花、起司、泡菜、秋葵）。
  4. 套餐組合：依訂單準備飲料（可樂）與附餐（味噌湯/小菜），組合成完整托盤。

（四）核心勝負機制：時間競速

  1. 倒數計時： 每一局遊戲開始時，系統會給予固定的初始時間（例如：90秒）。
  2. 加時獎勵： 當玩家成功將正確的餐點送達給客人時，系統將根據訂單難度給予「時間獎勵」（例如：一般牛丼 +10秒，套餐 +15秒）。
  3. 失敗懲罰： 若送錯餐點或讓客人等待過久導致憤怒離開，將會扣除時間作為懲罰。
  4. 遊戲結束： 當上方倒數計時歸零時，遊戲立即結束並進行分數結算。

（五）單人模式關卡進階設計

單人模式類似於《Papa's Freezeria》的職人體驗，重點在於挑戰極限操作。

  • 初期（Day 1-3）： 客人頻率低，初始時間長，讓玩家熟悉操作。
  • 中期（Day 4-7）： 客人生成速度加快，加時獎勵減少，容錯率降低。
  • 後期（Day 8+）： 進入「尖峰時刻（Rush Hour）」狀態，客人蜂擁而至，考驗玩家在四個工作站間的極限切換速度。

（六）關卡難度與訂單升級

為了避免遊戲後期僅是單純的手速比拚，訂單系統將隨著關卡推進而產生「變異」：

  • 基礎訂單： 標準的牛丼（飯+肉+所有配料）與標準套餐。
  • 客製化訂單（奧客模式）： 隨著遊戲進行，客人會提出特殊要求。
    • 範例 A： 「挑食者」 - 指定不要蔥花或不要起司（若玩家手快誤加，需倒掉重做）。
    • 範例 B： 「大胃王」 - 要求「特大盛」或「雙倍肉量」。
    • 範例 C： 「趕時間」 - 該名客人的等待紅條扣減速度為一般的 1.5 倍。

伍、評分與結算機制

為了貫徹「專業經營」的創業精神，本遊戲摒棄傳統的單純計分，改採用真實餐飲業的財務報表與人效指標進行結算。

（一）核心數據定義

1. 淨利潤 (Net Profit)

{淨利潤} = {營業額} - {食材成本} - {耗損}

  • 營業額： 成功賣出的餐點總價（例：牛丼$100、套餐$150）。
  • 食材成本： 每製作一份餐點的固定成本（例：每賣出一碗牛丼，系統自動扣除 $30 原料費）。
  • 耗損 (Loss)： 最關鍵的懲罰機制。任何因做錯、煮焦而丟進垃圾桶的食材，將以「售價的 100%」作為罰款。

2. 人時接客數 (CPH)

{人時接客數} = {總來客數} / {總工時}

系統會計算「完成訂單數」除以「遊戲時間 x 玩家人數」。數值越高，代表團隊手腳越快、默契越好。

（二）多人模式評估：年度績效考核

系統根據「淨利潤」與「人時接客數」的綜合表現，給予團隊一個諷刺性的企業職稱：

  • S 級（餐飲大亨）： 淨利極高 + 高人效。
  • A 級（黃金店長）： 利潤優秀。
  • B 級（資深社畜）： 表現平平。
  • C 級（薪水小偷）： 利潤低或耗損高。
  • F 級（惡意倒閉）： 負利潤。
  • MVP 機制： 結算時會抓出「最佳員工」與「廚房破壞者」。

（三）單人模式評估：分店營運目標

  • 過關條件： 每一關設有「目標淨利 (Target Profit)」，玩家必須在時間結束時，確保利潤大於目標值，否則視為「經營不善，店鋪倒閉」。
  • 星級評價：
    • 1星：剛好達到目標淨利。
    • 2星：超過目標淨利 120%。
    • 3星：超過目標淨利 150% 且人時接客數達到高標。

陸、遊戲畫面風格

圖1-1：主選單

打開游戲的第一個畫面，右下角為“開始游戲”按鈕，左下角為“退出游戲”按鈕。點擊“開始游戲”將引導玩家到下一個游戲畫面，點擊“退出游戲”則會關閉游戲。

圖1-2：取名字

玩家將從上一個游戲畫面引導到此畫面，玩家需要在開始游戲前取個名字，玩家輸入完名字之後在鍵盤點擊“ENTER”即可進入下一個畫面。

圖1-3：聯機畫面

玩家進入到此畫面後即可與好友分享房號邀請好友加入游戲房間，凑夠人數之後即可點擊“開始”按鈕。玩家們將隨機被分配到不同游戲站點。

圖1-4：製作牛丼

此畫面為製作牛丼畫面。玩家需根據訂單指示，操作左側的飯鍋與中間的肉鍋。

圖1-5：加工牛丼

此畫面為加工牛丼畫面，玩家需根據單子需求添加牛丼配料。配料選擇有葱花（左上），起司（左下），泡菜（右上），秋葵（右下）。點擊相應配料即可添加至牛丼上方。完成後即可滑動牛丼至下一個游戲畫面。

圖1-7：組合套餐

此畫面為組合套餐畫面，玩家需要根據單子需求將主餐與副餐組合在一起。玩家需將左上製作完成的牛丼與右上角的飲料與配菜組合到下方的托盤中，將單子從“單子區”滑動下來並貼在托盤上表示完成組合。完成後即可長按托盤並滑動至下一個玩家。

圖1-8：送餐給客人

此畫面為客席區，畫面上半部為客人的位置，下半部為組合好的餐點。玩家需點擊客人旁的圖標按鈕對客人做點餐或結賬動作。此外也需要將下半部組合好的餐點送至對應的客人。

柒、可行性分析與技術挑戰

（一）程式開發挑戰

  • 多人連線同步 (Multiplayer Networking)： 由於遊戲核心為 5 人即時協作，任何延遲都會破壞體驗。需使用 Photon 或類似的 Unity 連線套件，解決封包同步問題。
  • 食材狀態管理： 需即時運算 B、C、D 三個站點的庫存量，並與 E 玩家的補給動作進行邏輯綁定。

（二）美術資產需求

  • 需繪製 5 種不同的工作站背景。
  • 需繪製多種食材的動態變化（例如：生飯 -> 熟飯，滿鍋肉 -> 空鍋）。
  • UI 需包含清晰的動態提示（如：庫存不足的閃爍紅燈）。

（三）音效製作

  • 需製作或尋找所有界面下的背景音樂。
  • 需製作或尋找不同按鈕與操作的音效。

捌、程式架構與技術實作

本專案使用 Unity 引擎開發，語言為 C#。程式開發重點在於實現高精度的「多人即時連線同步」以及流暢的「物理手勢操作」。

（一）連線與同步架構 (Networking Architecture)

採用 Photon Unity Networking (PUN 2) 或 Mirror 作為連線解決方案：

  • 大廳與配對系統： 實作「開房」與「加入房間」邏輯。
  • 身份分配： 當 5 名玩家進入房間後，分配 A(點單)、B(煮飯)、C(加料)、D(組裝)、E(補貨) 五種不同的權限。
  • 物件狀態同步： 使用 PhotonView 或 NetworkIdentity 監聽食材狀態。透過 RPC 廣播，確保 E 玩家看到的是正確的畫面。

（二）核心互動系統

  • 拖曳與吸附 (Drag & Drop)： 實作 IDragHandler，讓食材能跟隨滑鼠/手指移動，並有自動吸附修正 (Snapping)。
  • 視角切換 (單人)： 透過 UI 按鈕或滑動，控制 Cinemachine 虛擬攝影機在四個工作站間移動。

（三）遊戲邏輯與數據運算

  • 訂單生成器： 建立資料庫，並實現權重隨機 (Weighted Random) 生成訂單。
  • 經濟運算： 即時運算 Score = Revenue - Cost - Loss。
  • 視覺動態： 使用 DOTween 處理 UI 彈跳與食材滑動效果。
